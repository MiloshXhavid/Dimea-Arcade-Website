## Summary

Phase 04 covers all the markup, assets, and metadata files that make the site discoverable, shareable, and correctly represented when linked in Discord, Twitter/X, or search results. The project is a flat static HTML site (no build step, no npm) deployed on Vercel — this is the simplest possible context for SEO work: everything is hand-edited directly in the HTML `<head>`, and static files are dropped at the repo root.

The two deliverable groups map cleanly to the two plans. Plan 04-01 handles pure-HTML metadata: `<title>`, meta description, canonical URL, Open Graph tags, Twitter card tags, JSON-LD structured data, `sitemap.xml`, and `robots.txt`. Plan 04-02 handles binary/generated assets: the SVG favicon with ICO fallback, the 180×180 apple-touch-icon, and the 1200×630 OG image (a static PNG created once, not generated at runtime).

The biggest planning concern for this phase is the OG image. It cannot be generated by a build tool (there is none) and needs to look like the brand. The recommended approach is to create it once as a static 1200×630 PNG using a design tool (Figma, Canva, or literally an HTML file opened in Chrome and screenshotted). Store it as `assets/og-image.png`. This is a manual creative task, not a code task — the plan must account for it.

**Primary recommendation:** Author all meta/OG/JSON-LD directly in `<head>` of every HTML file, drop static `sitemap.xml` and `robots.txt` at the repo root, and produce the OG image as a one-time static PNG. No tooling needed.

## Standard Stack

### Core

| Asset/Tag | Format/Value | Purpose | Why Standard |
|-----------|-------------|---------|--------------|
| `<title>` | 50–60 chars | Page title in search results and tabs | Google truncates at ~60 chars |
| `<meta name="description">` | 140–160 chars | Search result snippet | Ignored beyond 160 chars |
| `<link rel="canonical">` | Absolute HTTPS URL | Prevent duplicate content | Required on every page |
| `<meta property="og:*">` | Open Graph protocol | Social link previews | Universal: Discord, Slack, iMessage, Facebook |
| `<meta name="twitter:card">` | `summary_large_image` | Twitter/X previews | Falls back to OG tags if twitter: absent, but explicit is safer |
| JSON-LD `<script type="application/ld+json">` | SoftwareApplication | Google rich results for plugin | Google's preferred structured data format |
| `sitemap.xml` | Static XML | Tell crawlers which pages to index | Required for search coverage |
| `robots.txt` | Static text | Crawl permissions | Universal crawler directive |
| `favicon.ico` | ICO 32×32 | Legacy browser fallback | All browsers understand ICO |
| `icon.svg` | SVG | Modern browsers, scales perfectly | Supports dark mode via CSS media query |
| `apple-touch-icon.png` | PNG 180×180 | iOS home screen bookmark icon | Required by iOS Safari |
| `og-image.png` | PNG 1200×630 | OG/Twitter preview image | 1200×630 is the universal standard across platforms |

### Supporting

| Asset | Format/Value | Purpose | When to Use |
|-------|-------------|---------|-------------|
| `manifest.webmanifest` | JSON | Android/PWA icon references | Only needed if treating site as PWA; optional for this project |
| `icon-192.png` | PNG 192×192 | Android home screen | Only needed if including manifest |
| `og:image:width` / `og:image:height` | 1200 / 630 | Helps parsers pre-size image | Always include with og:image |
| `og:image:alt` | string | Accessibility + platform fallback | Include for completeness |
| `og:image:type` | `image/png` | MIME hint for parsers | Explicit is always better |

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| Static PNG OG image | Server-generated via Vercel Edge functions | Edge functions require framework (Next.js); not available for flat static sites |
| Static `sitemap.xml` | Auto-generated via build script | No build step exists; hand-authored sitemap for 5 pages is simpler |
| SVG favicon | ICO-only | ICO cannot adapt to dark mode; SVG is modern best practice with ICO as fallback |

**Installation:** None. No npm packages. No build tools.

### Pattern 3: SVG Favicon with Dark Mode

**What:** An SVG file containing the teal arc logo shape with an embedded `<style>` block for dark mode adaptation.
**Source:** https://blog.tomayac.com/2019/09/21/prefers-color-scheme-in-svg-favicons-for-dark-mode-icons/

```svg
<!-- icon.svg -->
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 32 32">
  <style>
    :root { --teal: #2ae8c4; }
    @media (prefers-color-scheme: dark) {
      :root { --teal: #2ae8c4; }
    }
  </style>
  <!-- Teal arc: right half of a circle, no left border -->
  <path d="M16 4 A12 12 0 0 1 16 28" fill="none"
        stroke="var(--teal)" stroke-width="3"
        stroke-linecap="round"/>
  <!-- Three dots stacked bottom-left -->
  <circle cx="8" cy="22" r="4" fill="white"/>
  <circle cx="8" cy="17" r="3" fill="rgba(255,255,255,0.7)"/>
  <circle cx="8" cy="13" r="2" fill="rgba(255,255,255,0.45)"/>
</svg>
```

Note: The exact SVG path for the teal arc needs to match the DOM logo in index.html (`.logo-arc` div is a CSS-border-radius arc, not a true SVG path). The planner should treat creating the SVG as a precise creative task, not a copy-paste.

### Pattern 4: sitemap.xml (Static)

**What:** Hand-authored XML file listing all public pages.

```xml

  </url>
  <url>
    <loc>https://dimea.audio/terms.html</loc>
```

### Pattern 5: robots.txt (Static)

**What:** Simple directive to allow all crawlers and point to sitemap.

```
User-agent: *
Allow: /

Sitemap: https://dimea.audio/sitemap.xml
```

### Anti-Patterns to Avoid

- **Relative URLs in og:image:** OG images MUST use absolute HTTPS URLs. `assets/og-image.png` will not work — must be `https://dimea.audio/assets/og-image.png`.
- **Applying same description to all pages:** Each page (privacy, terms, refund) should have a unique, appropriate meta description — not the same plugin pitch.
- **Skipping og:image dimensions:** Some parsers (Slack, iMessage) rely on explicit `og:image:width` and `og:image:height` to avoid layout reflows.
- **Using `twitter:image` with relative URL:** Same rule as og:image — must be absolute.
- **Putting JSON-LD on legal pages:** Structured data for the plugin belongs only on index.html. Legal pages don't benefit from SoftwareApplication schema.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| OG image generation | Canvas API code in a `<script>` tag | Static pre-generated PNG in assets/ | Runtime canvas generation is fragile, has CORS issues, and social crawlers (Googlebot, Discordbot) don't execute JS reliably |
| Sitemap generation | JS script that walks the directory | Hand-authored `sitemap.xml` | 5 pages — the file is 20 lines; automation adds zero value |
| Favicon multi-size generation | Manual Photoshop/Paint export at every size | favicon.io or Inkscape export | ICO multi-size format is complex to create correctly by hand |

**Key insight:** Social media crawlers often don't execute JavaScript. Everything meaningful for link previews (OG tags, JSON-LD) must be present in the raw HTML — not injected by script at runtime.

## Common Pitfalls

### Pitfall 1: Absolute URL Requirement for OG Images

**What goes wrong:** `og:image` set to `/assets/og-image.png` or `assets/og-image.png` — Discord and Twitter show no preview image.
**Why it happens:** The OG spec requires a fully-qualified URL because the crawler fetching the tag does so independently, not from the perspective of the page's host.
**How to avoid:** Always use `https://dimea.audio/assets/og-image.png` — the full domain + path.
**Warning signs:** Preview image missing in Discord's link embed or Twitter card validator.

### Pitfall 2: Domain Not Confirmed Yet

**What goes wrong:** All canonical URLs, OG:URL, og:image, and sitemap `<loc>` values need the real domain. Phase 05 (domain purchase) hasn't happened yet.
**Why it happens:** Phase ordering — SEO markup is authored before the domain is live.
**How to avoid:** Use `https://dimea.audio/` as the target domain throughout (it's the stated preference). When Phase 05 confirms the domain, a global find-replace corrects everything. Alternatively, use a Vercel preview URL as a temporary placeholder.
**Warning signs:** Canonical tags pointing to a domain that doesn't resolve yet — crawlers may penalise or ignore.

### Pitfall 3: OG Image Size / Crop Differences

**What goes wrong:** Brand text or logo placed near edges of the 1200×630 image gets cropped on some platforms (Discord crops to ~2:1 or shows letterboxed; Twitter shows slightly different viewport).
**Why it happens:** Platforms apply different crop boxes to the same image.
**How to avoid:** Keep all important content (logo, tagline, price) within a 1120×570 safe zone — 40px inset from each edge of the 1200×630 canvas.

### Pitfall 4: Legal Pages Have Wrong Meta Descriptions

**What goes wrong:** privacy.html, terms.html, refund.html all show "Play chords like a game..." in search results.
**Why it happens:** Copy-paste `<head>` from index.html without updating the description.
**How to avoid:** Each legal page needs a minimal but unique meta description:
- privacy.html: "DIMEA Arcade privacy policy — GDPR compliant, no cookie consent banner needed."
- terms.html: "DIMEA Arcade terms of service for Arcade Chord Control plugin and DIMEOLA hardware."
- refund.html: "DIMEA Arcade refund policy — 14-day no-questions return for plugin purchases."

### Pitfall 5: SVG Favicon Not Matching Brand Logo Exactly

**What goes wrong:** The SVG favicon looks like a rough approximation of the CSS-drawn logo, not the same shape.
**Why it happens:** The DOM logo uses CSS border-radius on a div, not an SVG path. Translating that to SVG requires careful measurement.
**How to avoid:** Open the live site, inspect `.logo-arc`, measure the exact border-radius / dimensions, and replicate as SVG arcs using the same proportions. Or: export the logo from a design tool that has it as a vector.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `<link rel="shortcut icon" href="favicon.ico">` | `<link rel="icon" href="/favicon.ico" sizes="32x32">` | ~2014 | `shortcut` prefix is obsolete; modern spec uses plain `rel="icon"` |
| 20+ favicon PNG files at different sizes | 3 files: favicon.ico, icon.svg, apple-touch-icon.png | 2021 (Evil Martians article; widely adopted) | SVG scales perfectly; browsers downscale; far less maintenance |
| `<meta name="twitter:*">` as primary | Twitter falls back to `og:*` | 2013+ (Twitter Card spec) | You can omit `twitter:*` entirely if og: tags are present; best practice is to include both |
| Inline `<script>` JSON-LD at page bottom | `<script type="application/ld+json">` in `<head>` | Google has accepted both, but `<head>` is conventional | No functional difference; `<head>` is cleaner |

**Deprecated/outdated:**
- `<meta name="keywords">`: Ignored by all major search engines since ~2009. Do not add.
- `<link rel="shortcut icon">`: Obsolete. Use `rel="icon"`.
- Multiple twitter-specific meta tags when og: equivalents exist: Twitter falls back automatically.

## OG Image Creation Guidance

The OG image is a creative asset, not a code task. Recommended approach for no-build static site:

**Option A (recommended): HTML + screenshot**
1. Create a temporary `og-image.html` file with brand colors, fonts (preloaded), logo mark, tagline, and price
2. Open in Chrome, set window to exactly 1200×630
3. Screenshot → save as `assets/og-image.png`
4. Delete the temporary HTML file

**Option B: Figma / Canva**
- Set canvas to 1200×630px
- Use brand colors: `#050810` background, `#2ae8c4` teal, `#e83a6e` magenta, `#e8ecf5` text
- Fonts: Syne (headings), Space Grotesk (body), DM Mono (labels)
- Export PNG

**Content to include on OG image:**
- DIMEA Arcade logo mark (the teal arc + dots)
- "DIMEA ARCADE" wordmark
- Primary tagline: "Bringing Harmony to the People"
- Sub-tagline: "Arcade Chord Control — VST Plugin — CHF 49"
- Keep all content within 1120×570 safe zone
